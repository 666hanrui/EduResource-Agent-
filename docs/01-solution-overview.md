# 01 · 总体方案（精简版）

> 第十五届"中国软件杯"A3 赛题 · EduResource-Agent

---

## 一、项目定位

**一句话定义**：面向高校计算机类课程，通过对话构建学习画像，由多个专业 Agent 协同生成个性化学习资源、规划学习路径并闭环优化的 AI Native 教育系统。

**它不是什么**：不是教育聊天机器人，不是题库系统，不是职业规划工具。

**它解决的核心问题**：学生面对海量教学资源时，无法找到匹配自己知识基础、学习风格和短板的内容。系统的输出不是"答案"，而是**为这个学生此刻定制生产的学习材料**。

---

## 二、核心思路与差异化

### 2.1 系统三层结构

```
输入层      画像构建（对话 + 上传 + 答题反馈）
   ↓
生产层      多 Agent 协作生成资源（核心）
   ↓
反馈层      评估 → 画像更新 → 下一轮推送变化
```

整个系统只有这三层。任何不属于这三层的功能都是干扰项。

### 2.2 与同类项目的差异化（35% 创新分核心）

聚焦 **2 个杀手锏**，不做大而全：

**杀手锏一：Agent 协作时序可视化**
前端实时呈现各 Agent 工作状态（甘特图式），让评委直接看到"画像 Agent 完成 → 文档 Agent 启动 → 题目 Agent 并行进行"。这是与"单 LLM 包装"项目的一眼区分。

**杀手锏二：可解释的个性化推送**
每份生成的资源下方显示"**为什么推荐给你**"：基于画像哪几个维度、哪些历史答题数据。点击可展开溯源链路（画像维度 → 短板分析 → 资源参数 → 生成结果）。突破 LLM 黑盒。

### 2.3 示范课程与杀手锏知识点

**示范课程**：数据结构（知识层级清晰、代码可跑、便于可视化）。
**杀手锏知识点**：链表与二叉树。系统在这两个知识点上做到极致——讲解文档、思维导图、可交互动画、自适应难度题目、Python/Java 双语代码案例全部联动一致。
**其他知识点**作为广度展示，资源生成完整但不强求深度。

## 三、学习画像设计

### 3.1 画像维度（8 维）

| 维度 | 用途 |
|------|------|
| 专业背景 | 决定术语深度 |
| 知识掌握度 | 决定起点和难度 |
| 学习目标 | 决定路径终点 |
| 学习风格 | 决定资源形态（图解/代码/推导） |
| 易错点 | 决定题目重点 |
| 资源偏好 | 决定优先生成什么 |
| 学习节奏 | 决定路径粒度 |
| 当前进度 | 决定推送时机 |

### 3.2 画像构建方式

- **首次构建**：自然语言对话，由 ProfileAgent 抽取并结构化
- **持续更新**：答题结果、资源停留时长、主动反馈
- **用户可见可改**：雷达图展示，支持手动修正
- **更新规则**：掌握度采用 `新值 = 旧值 × 0.7 + 本次表现 × 0.3` 的滑动更新

## 四、多智能体架构

### 4.1 Agent 设计（7 个，职责互不重叠）

| Agent | 职责 | 关键输出 |
|-------|------|----------|
| ProfileAgent | 画像抽取与更新 | 结构化画像 JSON |
| PlannerAgent | 知识点拆解 + 短板分析 + 资源规划 | 任务清单 |
| DocumentAgent | 生成讲解文档 | Markdown |
| ExerciseAgent | 生成自适应题目 | 题目 JSON + 解析 |
| CodeAgent | 生成代码案例 | 可运行代码 + 注释 |
| VisualAgent | 生成思维导图与动画数据 | Markmap / 动画步骤 JSON |
| EvaluationAgent | 答题分析 + 画像更新建议 | 评估报告 |

`PlannerAgent` 充当调度核心，不再单列 Orchestrator——简化协议、突出主线。
质量审核作为各生成 Agent 的内嵌环节，不单列 Agent。

### 4.2 协作模式

```
用户输入 → ProfileAgent
         ↓
       PlannerAgent（拆解任务、决定生成什么、并行还是串行）
         ↓
   ┌─────┼─────┬──────┬──────┐
Document  Exercise  Code  Visual
   └─────┴─────┴──────┴──────┘
         ↓
       前端流式展示 + 资源持久化
         ↓
学生答题 → EvaluationAgent → 更新画像
```

### 4.3 工程关键点

- 所有 Agent 输出使用 **Pydantic Schema** 约束，避免 LLM 输出格式失控
- Prompt 按 Agent 版本化管理（`prompts/document_agent_v1.md`）
- 每次调用记录 `task_id / agent / model / latency / token / status`，前端可查
- 流式输出（SSE）+ 异步任务（Celery 或 FastAPI BackgroundTasks）

## 五、资源生成（6 类）

| 资源类型 | 形式 | 实现方式 |
|----------|------|----------|
| 讲解文档 | Markdown | 讯飞星火 + Prompt 模板 |
| 思维导图 | Markmap | LLM 输出 Markdown 大纲 → Markmap 渲染 |
| 练习题 | JSON | 选择/填空/编程多类型 + 解析 + 易错点标注 |
| 代码案例 | Python + Java | 可运行代码 + 关键步骤注释 + 复杂度分析 |
| 教学动画 | SVG/Canvas | 前端渲染步骤化动画（链表插入、树遍历、图搜索） |
| 拓展阅读 | Link + 摘要 | RAG 检索课程资料 + LLM 摘要 |

**关于多模态视频**：主流程使用前端可视化动画（稳定、可交互、可暂停回放）。**演示视频中嵌入 2 段 SeeDance 预生成的教学短片**作为多模态展示亮点，规避现场调用风险。

## 六、学习路径与闭环

### 6.1 路径生成

PlannerAgent 综合以下因素输出阶段化路径：
- 知识点前置依赖关系
- 学生当前掌握度
- 每日可用时间
- 学习目标紧迫度
- 资源偏好

输出结构：`阶段 → 每日任务 → 关联资源 → 关联习题 → 阶段测试`。

### 6.2 闭环演示流程（演示视频核心 2 分钟）

1. 学生在链表插入题答错 2 次
2. EvaluationAgent 识别"指针修改顺序"为薄弱点
3. 画像中 `weakness` 字段更新
4. 系统自动降低下一题难度，并重新推送链表插入动画
5. 学生再次答题正确，画像掌握度回升
6. 路径中后续阶段任务自动调整

**这 2 分钟必须放进 7 分钟演示视频，是个性化最直接的证据。**

## 七、技术栈（明确表态）

| 层级 | 选型 | 理由 |
|------|------|------|
| **大语言模型** | **讯飞星火 X2 / 4.0 Turbo** | 赛题方推荐、中文能力、可争取 API 额度 |
| **多模态生成** | **SeeDance**（预生成短片） | 赛题点名工具，演示视频亮点 |
| Embedding | BGE-M3 | 开源中文效果好 |
| 向量数据库 | Qdrant | 课程资料检索 |
| 知识图谱 | Neo4j | 仅用于关键知识点关系约束生成（决赛增强） |
| 后端 | FastAPI + SQLAlchemy + Pydantic | 复用现有工程 |
| 前端 | React + TypeScript + Umi + Ant Design | 复用现有工程 |
| 可视化 | AntV G6 + Markmap + Monaco Editor + 自研 SVG 动画组件 | |
| 数据库 | 初赛 SQLite，决赛 PostgreSQL | 减少初赛部署复杂度 |
| 缓存 | Redis | 任务状态、SSE 会话 |
| 部署 | Docker Compose | 评委一键部署 |

## 八、实施范围（明确取舍）

### ✅ 初赛必做

- 学生端完整闭环（画像 → 资源 → 路径 → 答题 → 更新）
- 7 个 Agent 全部上线
- 6 类资源生成
- Agent 协作时序可视化
- 资源可解释性溯源
- 数据结构课程 9 个章节资源覆盖
- 演示视频 + PPT + 部署文档

### ⏸ 初赛不做

- 教师端、班级数据看板（决赛增强）
- 多课程扩展
- 知识图谱编辑器（仅用预置图谱）
- 实时 SeeDance 调用（用预生成资源替代）

### 🚀 决赛增强

- 教师端薄弱点诊断
- 知识追踪模型（BKT / DKT 替代规则更新）
- 多课程支持（数学、操作系统等）
- 真实多模态视频生成接入

## 九、风险与对策

| 风险 | 对策 |
|------|------|
| LLM 输出格式失控 | Pydantic Schema 强约束 + 失败重试 |
| 演示现场 API 不稳定 | 关键演示数据本地缓存，准备离线模式 |
| 多 Agent 工程复杂度 | 自研轻量调度器，不引入 LangGraph 等重框架 |
| 偏离赛题被判教育聊天 | 主导航无任何"对话"入口，所有交互围绕"资源" |
| 工作量超载 | 砍到 7 Agent / 6 资源类型，杀手锏只做 2 个知识点 |

## 十一、一句话总结

**用 7 个 Agent 把"学生说话 → 系统懂他 → 现场生产 → 答题反馈 → 系统懂得更准"这个闭环做到肉眼可见**——这就是我们和其他队的差别。
